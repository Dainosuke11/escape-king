# 【開発仕様書】オンライン対戦およびランクマッチ（RP）システムの実装
Firebaseを基盤とした「オンラインリアルタイム対戦機能」および「ランクマッチ（ランクポイント: RP）システム」を実装してください。仕様の詳細は以下の通りです。
## 1. システム構成・使用技術
 * **ユーザー認証（Firebase Authentication）:**
   * ユーザーの識別・データ紐付けに使用。手軽に開始できるよう、初期状態は**「匿名認証（Anonymous Authentication）」**をメインとしてください。
 * ** データベース（Cloud Firestore または Firebase Realtime Database）:**
   * マッチング待機列（キュー）の管理、および対戦中の盤面データのリアルタイム同期（2プロセス間の双方向通信）に使用します。
## 2. ランクの初期状態と解放条件
 * **初期ステータス:** 全プレイヤーは一律で「初心者」からスタートします。
 * **ランクマッチの解放条件:**
   * シングルプレイ（コンピューター対戦）の**レベル8**をクリアすること。
   * レベル8クリアのフラグが立った時点で、ステータスが「初心者」から**「ランク1（初期ポイント: 0 RP）」**に昇格し、オンラインのランクマッチモードが解放されます。
## 3. マッチング仕様および制限（不正防止）
オンライン対戦には、RPが変動する「ランクマッチ」と、変動しない「フレンド対戦」の2つのモードを設けます。
### ① ランクマッチのマッチングロジック
 1. **待機列（Queue）への登録:** プレイヤーがマッチングを開始すると、Firestoreの matching_queue コレクションに [ユーザーID, 現在のランク, 待機開始タイムスタンプ] を登録する。
 2. **段階的マッチング検索:**
   * 検索開始時は、**「自身の現在のランク ±1」**のプレイヤーを最優先で検索・マッチングする。
   * 10秒以上マッチングしない場合は検索範囲を**「±2」**に広げ、さらに20秒以上の場合は**「±3」**まで拡大する（最大差は±5までとし、それ以上離れたプレイヤーとはマッチングさせない）。
 3. **部屋の生成:** マッチングが成立したら、一意の room_id を作成し、双方のクライアントを該当の対戦部屋に遷移させ、先手・後手をランダムに決定する。
### ② 不正・ブースティング対策（重要）
 * **フレンド対戦の除外:** ルームIDを指定して遊ぶ「フレンド対戦（部屋を作って遊ぶ）」モードでは、試合結果によるRPの増減処理を一切行わない（is_ranked: false）。
 * **同一プレイヤーとの連戦制限:** ランクマッチ（ランダムマッチ）において、**直近2試合以内に戦った同一のユーザーID**と再度マッチングした場合は、その試合でのRP増減を無効（双方±0 RP）とする。
## 4. ランクポイント（RP）の増減ロジック
試合終了時、対戦相手とのランク差および現在のランク帯に応じて、以下のルールでRPの計算・更新を行ってください。ベースとなる変動値は **10ポイント** です。
### ① 通常帯（ランク11以上）の計算ルール
ランク差を D（対戦相手のランク - 自分のランク）と定義します。
| 対戦相手の条件 | 勝利時の処理 | 敗北時の処理
| **同格** (D = 0) | **+10 RP** 固定 | **-10 RP** 固定 |
| **格上** (D > 0) | **10 + (D × 2) RP** を獲得
※実力差があるほど多く貰える | **-10 RP** 固定
※格上に負けてもペナルティは増えない |
| **格下** (D < 0) | **10 - ( |  |
### ② 低ランク帯（ランク1〜10）のデフレ対策ルール
プレイヤー全体の総ポイント数が枯渇し、ランクが停滞する（デフレ）のを防ぐため、以下の救済措置（ポイントの新規生成）を組み込んでください。
 * **ランク1〜5:** 勝利時は通常通り加算。**敗北時の減少は「0 RP（マイナスなし）」**とする。
 * **ランク6〜10:** 勝利時は通常通り加算。**敗北時の減少は通常の半分「-5 RP（固定）」**とする。
## 5. ランクアップ・ランクダウン条件
 * **ランクアップ:** 各ランクにおいて、ポイントが **100 RP** に達するごとにランクが1つ上がります。（例: ランク1で100RPに達すると、ランク2の0RPになる。あふれたポイントはキャリーオーバーする）
 * **ランク上限:** **上限なし。** ランク99、ランク100以降も無限にカウントが上昇する設計にしてください。
 * **ランクダウン:** 敗北によって現在のランクの 0 RP を下回った場合、ランクが1つ下がります（90 RP に変動）。ただし、**ランク1の0RP未満には落ちない**ものとします。
## 6. 対戦中のリアルタイムデータ同期（ゲームループ）
Firestore上の rooms/{room_id} ドキュメント内に以下のデータを保持し、両クライアントでリアルタイムにリッスン（onSnapshot）して画面を同期させてください。
```json
{
  "room_id": "string",
  "status": "waiting / playing / finished",
  "player_host": "user_id_A",
  "player_guest": "user_id_B",
  "turn": "host / guest",
  "moves_left": 2,      // 1ターン内の残り移動回数(2〜0)
  "stones_left": 1,     // 1ターン内の残り石配置回数(1〜0)
  "host_king_pos": [x, y],
  "guest_king_pos": [x, y],
  "board_stones": [     // 石が置かれている座標の配列
    {"x": 3, "y": 5},
    {"x": 7, "y": 2}
  ],
  "winner": "null / user_id"
}

```
 * **更新のトリガー:** どちらかのプレイヤーが行動（移動、または石設置）を確定するたびに、データベースの該当項目をアップデートし、即座に相手の画面に反映させること。
 * **切断処理:** 試合中にどちらかの通信が5秒以上途切れた場合、またはブラウザが閉じられた場合は、残された側の「不戦勝（+10 RP）」として処理するセーフティを設けてください。
以上に基づき、バックエンドの構築およびフロントエンドへのロジックの組み込みをお願いいたします。


## 1. 発生している問題と原因
 * **現象:** 通信が切れると「接続が切れました」と表示されるだけで、その後ゲームに復帰できない。
 * **原因:** 1. クライアント側の wsConn.onclose 内で再接続ロジックが走っておらず、その後の wsSend() が失敗して即座に完全に切断されてしまうため。
   2. サーバー側（ONLINE LOBBY UI等）で、再接続してきたプレイヤーが「元の試合のプレイヤーと同じである」と判定するロジック（セッション保持）がないため。
## 2. クライアント側（ブラウザ/フロントエンド）の修正要求
### ① 自動再接続（リトライ）ロジックの追加
 * wsConn.onclose が発生した際、即座にエラー終了とせず、数秒おきに自動で再接続を試みる reconnect() 関数を実装してください。
 * **仕様案:** * 3秒ごとに最大5回（計15秒間）再接続を試行する。
   * 再接続に成功した場合は、自動的に元のゲーム状態への復帰処理（後述の再認証メッセージ送信）を行う。
   * 最大回数試行しても繋がらない場合のみ、最終的な「切断エラー」を画面に表示する。
### ② 再接続中のUI（ONLINE LOBBY UI）の制御
 * 通信が切断されて再接続を試みている間は、画面に「通信が切断されました。再接続しています...（残り○秒）」といったダイアログやインジケータを表示し、ユーザーが操作できないように（誤った wsSend が走らないように）制御してください。
## 3. サーバー側（バックエンド/ルーム管理）の修正要求
### ① 一時切断（保留）ステータスの実装
 * 対戦中のプレイヤーの接続が切れた（WebSocketがクローズした）際、**即座にルームを解体したり、相手の不戦勝にしたりしないでください。**
 * ルームのステータスを「一時切断（Suspended）」とし、**15秒〜30秒程度の猶予時間（タイマー）**を開始してください。この猶予時間内にプレイヤーが戻ってくれば試合を再開します。タイマーがゼロになった場合のみ、残されたプレイヤーの不戦勝として処理してください。
### ② 同一プレイヤーの識別（セッション再開判定）
 * プレイヤーが再接続してきた際、新しく入ってきた人ではなく「さっきまで戦っていた人」だと判定する仕組みを入れてください。
 * **実装案:** * 初回接続時にサーバーから各プレイヤーへ一意の「セッションID（またはトークン）」を発行し、クライアントの localStorage 等に保存させる。
   * 再接続時、クライアントはまずそのセッションIDをサーバーに送信する。
   * サーバー側は、受け取ったセッションIDが「一時切断中」のルームのプレイヤーと一致した場合、新しいWebSocket接続を既存のルームのプレイヤー枠に再紐付け（紐付け直し）する。
### ③ 最新盤面データの同期
 * 再接続・再紐付けが成功した直後、サーバーからそのプレイヤーに対して「現在の最新の盤面データ（各コマの位置、ターン、残り時間など）」を送信し、クライアント側の画面を切断前の状態に完全に復元（同期）させてください。
以上、通信の瞬断に強いオンライン対戦環境へのアップデートをお願いいたします。
